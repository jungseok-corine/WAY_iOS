## 슬라이드 3: 아키텍처

### Clean Architecture + MVVM

```
┌─────────────────────────────────────────────────┐
│  Presentation Layer                             │
│  ┌────────────┐  ┌───────────┐  ┌───────────┐  │
│  │ View       │──│ ViewModel │──│ UseCase   │  │
│  │ Controller │  │           │  │ (Domain)  │  │
│  └────────────┘  └───────────┘  └─────┬─────┘  │
├───────────────────────────────────────┼─────────┤
│  Domain Layer                         │         │
│  ┌──────────────┐  ┌────────────────┐ │         │
│  │ Entity       │  │ UseCase        │←┘         │
│  │ (비즈니스 모델)│  │ (비즈니스 로직) │           │
│  └──────────────┘  └───────┬────────┘           │
├────────────────────────────┼────────────────────┤
│  Data Layer                │                    │
│  ┌────────────┐  ┌────────┴───────┐             │
│  │ API (Moya) │──│ Repository     │             │
│  │ Service    │  │ (데이터 소스 추상화)│          │
│  └────────────┘  └────────────────┘             │
│  ┌──────────────────────────────┐               │
│  │ Models (Request / Response)  │               │
│  └──────────────────────────────┘               │
└─────────────────────────────────────────────────┘
```

**포트폴리오 어필 포인트**:
- Protocol 기반 의존성 주입으로 각 레이어 간 결합도를 낮춤
- UseCase를 Protocol + Impl로 분리하여 테스트 가능한 구조 확보
- Repository 패턴으로 데이터 소스를 추상화하여 비즈니스 로직과 네트워크 로직 분리

## 슬라이드 4: 담당 기능 — 피드 CRUD

### 구현 내용
피드 생성(Create), 조회(Read), 수정(Update), 삭제(Delete) 전체 플로우를 구현

### 주요 구현 사항

**피드 작성 (Create)**
- 일정 선택 드롭다운 (날짜별 그룹핑, 페이지네이션)
- PHPickerViewController를 활용한 다중 이미지 선택 (최대 10장)
- DispatchGroup을 활용한 비동기 이미지 로딩
- Multipart/form-data를 통한 이미지 + JSON 복합 업로드

**피드 목록 조회 (Read)**
- UITableView 기반 피드 리스트
- 무한 스크롤(Infinite Scroll) 페이지네이션 구현
- 가로 스크롤 이미지 갤러리 + 페이지 인디케이터
- "더 보기" 텍스트 확장/축소 기능

**피드 수정 (Update)**
- 기존 이미지 URL을 Kingfisher로 로딩 후 편집 화면에 표시
- 이미지 추가/삭제 후 서버와 동기화
- 제목/내용 수정 반영

**피드 삭제 (Delete)**
- 확인 Alert 후 삭제 처리
- 삭제 후 로컬 배열 즉시 업데이트 (Optimistic UI)

### 기술적 하이라이트
- **Multipart 이미지 업로드**: `MultipartFormDataHelper`로 Request body(JSON) + 이미지(JPEG)를 함께 전송
- **무한 스크롤**: `scrollViewWillBeginDecelerating`에서 offset 계산 후 다음 페이지 로드
- **복합 셀 구성**: FeedsTableViewCell 안에 프로필 뷰, 이미지 갤러리(CollectionView), 북마크 버튼, 옵션 메뉴를 조합

### 구현 내용
피드를 북마크(즐겨찾기), 숨김(아카이브) 처리하고 마이페이지에서 관리

### 주요 구현 사항

**피드 숨김 (Hide/Archive)**
- 피드 숨김 처리 및 복원 기능
- 작성자/비작성자에 따른 옵션 메뉴 분기 처리

### 기술적 하이라이트

## 슬라이드 6: 담당 기능 — 로그인 / 회원가입

### 구현 내용
이메일 로그인, 카카오 로그인, Apple 로그인 3가지 방식의 인증 시스템 구현

### 주요 구현 사항

**이메일 로그인**
- 이메일/비밀번호 입력 후 JWT 토큰 기반 인증
- 로그인 성공 시 AccessToken, RefreshToken, FCM Token 저장
- `AutoTokenPlugin`(Moya Plugin)으로 모든 API 요청에 Bearer Token 자동 주입

**카카오 로그인**
- KakaoSDK 연동: 카카오톡 앱/웹 로그인 분기 처리
- 신규 회원(에러코드 KN007) 감지 시 회원가입 화면으로 자동 전환

**Apple 로그인**
- ASAuthorizationAppleIDProvider 활용
- Identity Token 기반 서버 인증
- 최초 로그인 시 회원가입 플로우 연결

**회원가입**
- 이용약관 동의 → 이메일 인증 → 정보 입력 → 가입 완료 (4단계 플로우)
- 이메일 인증코드 발송 + 5분 타이머 카운트다운
- 정규식 기반 실시간 유효성 검사 (이메일, 비밀번호, 한글 이름 1~4자)

**비밀번호 찾기**
- 이메일 인증 후 비밀번호 재설정
- 비밀번호 일치 여부 실시간 확인

### 기술적 하이라이트
- **Moya Plugin 활용**: `AutoTokenPlugin`으로 인증 토큰 관리를 네트워크 레이어에서 일괄 처리
- **소셜 로그인 에러 핸들링**: 에러 코드 기반으로 신규/기존 회원 분기 처리
- **유효성 검사 모듈화**: `ValidationHelper`에 정규식 로직을 집중하여 재사용성 확보
- **토큰 갱신**: `TokenReissueUseCase`를 통한 토큰 자동 갱신 구조

## 슬라이드 8: 담당 기능 — 마이페이지 / 회원탈퇴

### 구현 내용
프로필 관리, 메뉴 네비게이션, 회원 탈퇴 플로우 구현

### 주요 구현 사항

**마이페이지**
- 프로필 이미지 변경 (PHPickerViewController + Multipart 업로드)
- 사용자 정보(이름, 이메일) 표시
- 태그 기반 라우팅으로 6개 하위 메뉴 연결
  - 내 정보 관리 / 위치 즐겨찾기 / 피드 보관함 / 피드 숨김 / 공지사항 / 문의하기

**회원탈퇴 (4단계)**
1. 탈퇴 동의 → 2. 탈퇴 사유 입력 → 3. 비밀번호 확인 → 4. 탈퇴 완료
- 로그인 타입(일반/카카오/Apple)에 따른 분기 처리
- 탈퇴 완료 시 UserDefaults 전체 초기화 후 로그인 화면으로 이동

**로그아웃**
- 서버 로그아웃 API 호출 + 로컬 토큰 삭제

### 기술적 하이라이트
- **다단계 플로우 관리**: 4단계 화면 전환을 일관된 ViewModel로 관리
- **로그인 타입별 분기**: 일반/소셜 로그인에 따른 탈퇴 처리 분기
- **데이터 정리**: 로그아웃/탈퇴 시 UserDefaults, 토큰 등 완전 초기화

## 슬라이드 10: 네트워크 레이어 설계

### Moya 기반 API 레이어

```
API Enum (TargetType)
  ↓
Service (Protocol + Impl)
  ↓
Repository (데이터 소스 추상화)
  ↓
UseCase (비즈니스 로직)
  ↓
ViewModel → View
```

### 포트폴리오 어필 포인트

**Moya TargetType 활용**
- Enum case로 엔드포인트를 타입 안전하게 관리
- Content-Type 분기: JSON / Multipart/form-data 자동 판별

**AutoTokenPlugin (인증 미들웨어)**
- Moya의 PluginType을 활용하여 모든 요청에 Bearer Token 자동 삽입
- 토큰 갱신 로직과 분리된 단일 책임 구조

**에러 핸들링**
- `GenericResponse<T>` 래퍼로 서버 응답 통일
- `APIResponseHandler`로 응답 파싱 일괄 처리
- `APIError` enum으로 에러 타입 중앙 관리

## 슬라이드 11: 트러블슈팅 / 기술적 도전 (예시)

> (아래 내용 중 실제 경험한 것을 골라서 작성하세요)

### 1. Multipart 이미지 업로드 처리
- **문제**: JSON Request Body와 이미지 파일을 동시에 전송해야 하는 상황
- **해결**: `MultipartFormDataHelper`를 구현하여 Request Body를 JSON Data로, 이미지를 JPEG Data로 변환한 뒤 Moya의 MultipartFormData Task로 통합 전송
- **결과**: 피드 작성/수정 시 최대 10장의 이미지를 안정적으로 업로드

### 2. 소셜 로그인 에러 핸들링
- **문제**: 카카오/Apple 로그인 시 신규 회원과 기존 회원을 구분해야 하는 상황
- **해결**: 서버 에러 코드(KN007)를 감지하여 신규 회원일 경우 자동으로 소셜 회원가입 화면 전환
- **결과**: 사용자가 별도 조작 없이 자연스러운 가입 플로우 경험

### 3. 무한 스크롤 페이지네이션
- **문제**: 피드 목록에서 대량의 데이터를 효율적으로 로드해야 하는 상황
- **해결**: UIScrollViewDelegate의 스크롤 위치 계산 + ViewModel의 page 카운터를 활용한 점진적 데이터 로딩
- **결과**: 끊김 없는 스크롤 경험 + 불필요한 네트워크 요청 최소화

### 4. 토큰 기반 인증 관리
- **문제**: 모든 API 요청에 인증 토큰을 일일이 주입하는 반복 코드 발생
- **해결**: Moya의 PluginType을 활용한 `AutoTokenPlugin`으로 인증 처리를 미들웨어 패턴으로 추상화
- **결과**: API 레이어에서 인증 로직을 완전히 분리하여 코드 중복 제거

## 참고: 프로젝트 폴더 구조

```
Where_Are_You/
├── Presentation/          ← View + ViewController + ViewModel
│   ├── Login/
│   │   ├── Login/         ← 이메일/소셜 로그인
│   │   ├── SingUp/        ← 회원가입 (이메일 인증 포함)
│   │   └── FindAccount/   ← 비밀번호 찾기
│   ├── Main/
│   │   ├── Home/          ← 메인 홈 (배너, D-Day, 피드)
│   │   ├── FriendFeed/
│   │   │   └── Feeds/     ← 피드 CRUD + 북마크/숨김
│   │   ├── Mypage/        ← 마이페이지 + 회원탈퇴
│   │   └── Location/      ← 위치 즐겨찾기
│   ├── SNS/               ← 소셜 로그인 회원가입
│   └── Notification/      ← 알림
├── Domain/                ← UseCase + Entity
│   ├── Entities/          ← Feed, Member, DDay 등
│   └── UseCases/          ← 기능별 비즈니스 로직
├── Data/                  ← API + Repository + Model
│   ├── Network/           ← Moya API 정의 + Service
│   ├── Repositories/      ← Repository 구현체
│   └── Models/            ← Request/Response DTO
└── Utils/                 ← 공통 유틸리티
```
